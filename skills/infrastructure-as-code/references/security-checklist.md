# IaC Security Checklist & Best Practices

This document serves as the source of truth for security, compliance, tagging requirements,
CI-native approval gates, state locking, and plan review artifact requirements governing all
Infrastructure-as-Code manifests generated by the `infrastructure-as-code` skill.

## Agent Self-Approval Prohibition (SEC-8.2)

| # | Rule | Severity |
|---|------|----------|
| 1 | The agent generating Terraform/K8s/CloudFormation manifests **MUST NOT** also approve the resulting plan. | CRITICAL |
| 2 | Approval MUST come from an **external human reviewer** or a **CI-native protected-environment gate** (e.g., GitHub Environment `required_reviewers: 1`). | CRITICAL |
| 3 | Workflows MUST separate the `plan` job from the `apply` job, with the `apply` job gated by an environment that requires external approval. | CRITICAL |
| 4 | If a user attempts to instruct the agent to auto-approve, the agent MUST refuse and re-present the plan review document. | CRITICAL |

## Mandatory Tags

Every taggable resource MUST include these four tags. Validation (Step 6 of the IaC workflow)
rejects any manifest missing them.

| Tag | Description | Example |
|-----|-------------|---------|
| `environment` | Deployment environment | `dev`, `staging`, `prod` |
| `owner` | Team or individual accountable | `team-platform`, `alice.smith` |
| `cost-center` | Billing allocation code | `cc-1234`, `eng-infra` |
| `project` | Project or service identifier | `payments-api`, `data-pipeline` |

**Enforcement**: Use provider `default_tags` (AWS), `labels` (GCP), or `tags` (Azure) blocks
so every child resource inherits them. Explicitly validate with `checkov` rule `CKV_AWS_18`
and custom tagging policies.

## Secret Management Rules

| # | Rule | Severity |
|---|------|----------|
| 1 | NEVER embed passwords, API keys, tokens, or certificates in `.tf`, `.yaml`, `.json`, or `.tfvars` files. | CRITICAL |
| 2 | NEVER commit `.tfstate`, `.tfstate.backup`, or `.tfvars` with sensitive values to version control. | CRITICAL |
| 3 | ALWAYS reference secrets via external vaults: HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, 1Password Secrets Automation. | HIGH |
| 4 | ALWAYS use environment variables or CSI secret drivers for runtime secret injection in Kubernetes. | HIGH |
| 5 | ALWAYS rotate secrets via vault mechanisms; never bake static credentials into golden images. | HIGH |
| 6 | Use Terraform `sensitive = true` on all output variables that expose credentials or connection strings. | MEDIUM |

**Vault integration patterns**:
- AWS: `data.aws_secretsmanager_secret_version.xxx.secret_string`
- GCP: `data.google_secret_manager_secret_version.xxx.secret_data`
- Azure: `data.azurerm_key_vault_secret.xxx.value`
- HashiCorp Vault: `data.vault_generic_secret.xxx.data["password"]`

## Encryption Requirements

| Resource | At-Rest | In-Transit | Notes |
|----------|---------|------------|-------|
| EBS / Compute disks | REQUIRED (KMS/CMEK) | N/A | `encrypted = true` + dedicated KMS key |
| RDS / Cloud SQL / Azure SQL | REQUIRED (KMS/CMEK) | REQUIRED (TLS) | Enable `performance_insights` for audit |
| S3 / GCS / Blob Storage | REQUIRED (AES-256 or KMS) | REQUIRED (TLS) | Block public access by default |
| ElastiCache / Memorystore / Redis | REQUIRED | REQUIRED (TLS) | Use auth tokens from vault |
| EKS / GKE / AKS secrets | REQUIRED (KMS) | REQUIRED (TLS) | Enable envelope encryption |
| Load Balancers | N/A | REQUIRED (TLS 1.2+) | ACM / Google-managed / Azure-managed certs |
| VPC Flow Logs / Audit Logs | REQUIRED (encrypted bucket) | N/A | Centralize to SIEM |

## Network Segmentation

| # | Rule | Severity |
|---|------|----------|
| 1 | Place compute workloads in **private subnets** with no public IPs. | HIGH |
| 2 | Use **NAT Gateway / Cloud NAT** for outbound internet from private subnets. | HIGH |
| 3 | Restrict security group / firewall rules to **minimum required ports and sources**. | HIGH |
| 4 | Isolate databases and caches in dedicated subnets; expose only to compute security groups. | HIGH |
| 5 | Enable **VPC Flow Logs** (AWS), **VPC Flow Logging** (GCP), or **NSG Flow Logs** (Azure). | MEDIUM |
| 6 | Use **WAF** or **Cloud Armor / Front Door** for internet-facing load balancers. | MEDIUM |
| 7 | Enable **DDoS Protection** (AWS Shield / GCP Cloud Armor / Azure DDoS) for public endpoints. | MEDIUM |

## Identity & Access (Least Privilege)

| # | Rule | Severity |
|---|------|----------|
| 1 | Create dedicated IAM roles / service accounts per workload; never reuse broad roles. | HIGH |
| 2 | Scope policy `Resource` ARNs to the specific project/environment prefix. | HIGH |
| 3 | Avoid `Action = "*"` and `Resource = "*"` in any policy. | HIGH |
| 4 | Enable **IAM database authentication** where supported (RDS IAM, Cloud SQL IAM). | MEDIUM |
| 5 | Require **MFA** for privileged human roles and break-glass access. | HIGH |
| 6 | Rotate IAM credentials and service account keys automatically; prefer IAM roles / workload identity. | HIGH |
| 7 | Use **AWS IAM Roles for Service Accounts (IRSA)** / **GCP Workload Identity** / **Azure Workload Identity** for pod-level credentials. | HIGH |

## Checkov Pre-Plan Gate & Rules Quick Reference

Checkov MUST be run on generated manifests **before** `terraform plan` is executed.
This is a hard gate: no plan artifact may be produced until Checkov passes the CRITICAL threshold.

Run `checkov --framework terraform -d .` (or `kubernetes`, `cloudformation`).
Critical rules to address before any plan or apply:

| Rule | Description | Severity |
|------|-------------|----------|
| CKV_AWS_18 | Ensure S3 bucket has access logging enabled | MEDIUM |
| CKV_AWS_19 | Ensure S3 bucket is encrypted | HIGH |
| CKV_AWS_20 | Ensure S3 bucket has MFA delete enabled | MEDIUM |
| CKV_AWS_21 | Ensure S3 bucket versioning is enabled | MEDIUM |
| CKV_AWS_23 | Ensure security groups do not allow unrestricted ingress | HIGH |
| CKV_AWS_40 | Ensure IAM policies are attached only to groups/roles | MEDIUM |
| CKV_AWS_79 | Ensure EC2 instances have detailed monitoring enabled | LOW |
| CKV_AWS_126 | Ensure RDS instance has deletion protection enabled (prod) | HIGH |
| CKV_AWS_130 | Ensure VPC flow logs are enabled | MEDIUM |
| CKV_AWS_144 | Ensure S3 bucket has cross-region replication | MEDIUM |
| CKV_AWS_157 | Ensure RDS instance has Multi-AZ enabled (prod) | HIGH |
| CKV_AWS_16 | Ensure RDS instance is encrypted | HIGH |
| CKV_AWS_17 | Ensure RDS instance uses IAM authentication | MEDIUM |
| CKV_K8S_21 | Ensure default namespace is not used | MEDIUM |
| CKV_K8S_38 | Ensure readiness/liveness probes are configured | MEDIUM |
| CKV_K8S_40 | Ensure containers do not run as root | HIGH |
| CKV_K8S_43 | Ensure containers have resource limits | HIGH |
| CKV2_AWS_11 | Ensure VPC flow logging is enabled in all VPCs | HIGH |
| CKV2_AWS_12 | Ensure IAM credentials are not hardcoded | CRITICAL |
| CKV2_AWS_19 | Ensure ECR repository is encrypted | HIGH |
| CKV_AZURE_2 | Ensure SQL server has threat detection enabled | MEDIUM |
| CKV_AZURE_23 | Ensure SQL database is encrypted | HIGH |
| CKV_GCP_6 | Ensure Cloud SQL database requires SSL | HIGH |
| CKV_GCP_20 | Ensure Cloud SQL is encrypted with CMEK | HIGH |
| CKV_GCP_32 | Ensure VPC flow logging is enabled | MEDIUM |

**Gate rules**:
- If any **CRITICAL** finding is reported, **HALT** the workflow and require remediation before plan generation.
- If any **HIGH** finding is reported, require documented risk acceptance or remediation before plan generation.
- The Checkov report (JSON or SARIF) MUST be included in the plan review artifact (`PLAN_REVIEW.md`).
- Re-running Checkov after apply is recommended for continuous compliance.

## Kubernetes Security Defaults

All Kubernetes manifests MUST pass the following checks:

| # | Rule | Severity |
|---|------|----------|
| 1 | Containers MUST NOT run as root (`runAsNonRoot: true`, `runAsUser: >=10000`). | HIGH |
| 2 | Containers MUST define both `requests` and `limits` for CPU and memory. | HIGH |
| 3 | Pods MUST use a dedicated ServiceAccount; never `default`. | MEDIUM |
| 4 | NetworkPolicies MUST restrict ingress/egress to required ports and namespaces. | HIGH |
| 5 | Secrets MUST be mounted via external secret operator (CSI driver, Vault Agent Injector). | HIGH |
| 6 | Liveness and readiness probes MUST be defined for every container. | MEDIUM |
| 7 | PodSecurityPolicies / Pod Security Standards (`restricted`) MUST be enforced. | HIGH |
| 8 | Container images MUST use immutable tags (digest or explicit version); never `latest`. | HIGH |
| 9 | ResourceQuota and LimitRange MUST be defined per namespace. | MEDIUM |
| 10 | Audit logging MUST be enabled at the cluster level. | MEDIUM |

**Validation tools**: `kubeconform`, `kubesec`, `checkov --framework kubernetes`,
`kubectl apply --dry-run=server`.

## State Locking & Drift Management

| # | Rule | Severity |
|---|------|----------|
| 1 | NEVER store Terraform state in version control. Use remote backend (S3 + DynamoDB / GCS / Azure Blob). | CRITICAL |
| 2 | ALWAYS enable state locking to prevent concurrent modifications. | CRITICAL |
| 3 | ALWAYS enable state encryption at rest (SSE-KMS / CMEK / Azure CMK). | HIGH |
| 4 | ALWAYS generate `terraform plan -out=plan.tfplan` before any apply; binary plan is required for apply. | HIGH |
| 5 | ALWAYS capture a state backup (`terraform state pull`) before destructive changes and store in a versioned backup location. | HIGH |
| 6 | Run `terraform plan` (or equivalent) before every apply to detect drift. | HIGH |
| 7 | Run periodic drift detection scans and alert on unmanaged changes. | MEDIUM |
| 8 | Maintain state backup snapshots before any production state migration. | HIGH |

### State Locking Configuration

**AWS (S3 + DynamoDB)**:
```hcl
terraform {
  backend "s3" {
    bucket         = "myorg-terraform-state"
    key            = "myproject/prod/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    kms_key_id     = "arn:aws:kms:us-east-1:123456789:key/..."
    dynamodb_table = "terraform-locks"
  }
}
```
- S3 bucket must have versioning, server-side encryption (SSE-KMS), and block public access.
- DynamoDB table must use `PayPerRequest` billing and have a partition key `LockID` (String).

**GCP (GCS)**:
```hcl
terraform {
  backend "gcs" {
    bucket  = "myorg-terraform-state"
    prefix  = "myproject/prod"
    encryption_key = "projects/.../locations/.../keyRings/.../cryptoKeys/..."
  }
}
```
- GCS bucket must have versioning, uniform access, and CMEK encryption.
- Locking is handled natively by GCS Object Administration.

**Azure (Blob Storage)**:
```hcl
terraform {
  backend "azurerm" {
    resource_group_name  = "terraform-state-rg"
    storage_account_name = "myorgtfstate"
    container_name       = "terraform-state"
    key                  = "myproject/prod/terraform.tfstate"
    use_azuread_auth     = true
  }
}
```
- Blob container must have versioning, soft-delete, and CMK encryption.
- Lease-based locking is implicit with the standard backend.

### State Backup Procedure
```bash
# Before any apply, especially in production
cd <terraform-directory>
terraform state pull > "state-backup-$(date +%s).json"
# Upload to separate versioned backup bucket/container
# Document backup location in PLAN_REVIEW.md
```

## Backup & Disaster Recovery

| # | Rule | Severity |
|---|------|----------|
| 1 | Production databases MUST have automated backups with `backup_retention_period >= 7`. | HIGH |
| 2 | Production databases MUST have `deletion_protection = true`. | HIGH |
| 3 | S3 / GCS / Blob buckets MUST have versioning and lifecycle policies. | MEDIUM |
| 4 | Cross-region replication MUST be enabled for critical data stores in production. | MEDIUM |
| 5 | Document rollback strategy in every `PLAN_REVIEW.md`. | HIGH |

## Cost & Resource Efficiency

| # | Rule | Severity |
|---|------|----------|
| 1 | Use right-sized instance types for the environment (dev = burstable, prod = dedicated). | MEDIUM |
| 2 | Enable auto-scaling for compute and database read replicas where applicable. | MEDIUM |
| 3 | Use spot / preemptible instances for non-critical dev/staging workloads. | LOW |
| 4 | Define lifecycle policies to transition or delete aged storage objects. | MEDIUM |
| 5 | Tag all resources with `cost-center` for chargeback visibility. | HIGH |
| 6 | Run `infracost` on every plan; flag >20% cost deltas for additional review. | MEDIUM |

## Production Environment Gates

Additional controls for `environment = prod`:

| Control | Requirement |
|---------|-------------|
| Approval | **External human or CI-native gate required** — agent cannot self-approve |
| CI-native Gate | GitHub Environment with `required_reviewers >= 1`, or GitLab manual job, or Azure DevOps environment approval |
| Multi-AZ | All stateful services (RDS, ElastiCache, Kafka) deployed across AZs |
| Deletion Protection | `deletion_protection = true` on all stateful resources |
| Encryption | CMEK/KMS for all storage and databases |
| Monitoring | CloudWatch/Stackdriver/Azure Monitor dashboards and alarms pre-configured |
| CI/CD Only | Apply operations MUST run through CI/CD pipeline, never local workstation |
| Peer Review | Plan review document requires second-person approval for prod |
| Change Window | Scheduled maintenance window for non-emergency changes |
| Rollback Plan | Documented rollback steps and rollback validation criteria in `PLAN_REVIEW.md` |
| Checkov Gate | Checkov scan MUST pass (no CRITICAL findings) before plan generation |
| State Backup | `terraform state pull` backup captured and stored before apply |

### CI-native Approval Gate Patterns

**GitHub Actions**:
```yaml
terraform-apply:
  needs: terraform-plan
  runs-on: ubuntu-latest
  environment:
    name: production
    url: ${{ steps.apply.outputs.url }}
  # Repository setting: Environment 'production' must have required_reviewers: 1
```

**GitLab CI**:
```yaml
apply:
  stage: deploy
  needs: [plan]
  when: manual
  only:
    - main
  environment:
    name: production
    on_stop: rollback
```

**Azure DevOps**:
```yaml
- stage: Apply
  jobs:
  - deployment: TerraformApply
    environment: 'production'
    strategy:
      runOnce:
        deploy:
          steps:
            - download: current
              artifact: plan.tfplan
            # apply steps...
```

**Enforcement**: The `apply` step/job MUST be separated from the `plan` step/job by an
environment or manual gate that requires a human outside the agent's execution context.
The agent MUST refuse to generate workflows that bypass this separation.

## Validation Command Reference

**Pre-Plan Security Gate (MUST run before `terraform plan`)**:
```bash
terraform fmt -recursive
terraform validate
tflint --format compact
checkov --framework terraform -d . --soft-fail false --check CRITICAL
# If CRITICAL findings exist, HALT and remediate before proceeding.
```

**Plan & Review Artifact Generation**:
```bash
terraform plan -out=plan.tfplan
terraform show -json plan.tfplan > plan.json
infracost breakdown --path . --format json > infracost.json
# Generate PLAN_REVIEW.md
python scripts/generate-plan-review.py \
  --plan-json plan.json \
  --checkov-json checkov.json \
  --infracost-json infracost.json \
  --project myapp --env prod --owner team-platform --cost-center cc-1234 \
  --output ./PLAN_REVIEW.md
```

**Post-Approval Apply**:
```bash
terraform apply plan.tfplan
```

**Kubernetes**:
```bash
kubeconform -strict -summary -output json .
kubesec scan *.yaml
checkov --framework kubernetes -d . --soft-fail false --check CRITICAL
kubectl apply --dry-run=server -f manifests/
# Require external approval before kubectl apply --server-side
```

**CloudFormation**:
```bash
cfn-lint template.yaml
cfn-guard validate --rules rules.guard --data template.yaml
checkov --framework cloudformation -d . --soft-fail false --check CRITICAL
aws cloudformation create-change-set --stack-name mystack --template-body file://template.yaml
# Require external approval before aws cloudformation execute-change-set
```
