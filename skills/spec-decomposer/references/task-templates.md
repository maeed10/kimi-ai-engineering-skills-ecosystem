# Task Node Templates for Obsidian

This reference defines the YAML frontmatter schema and markdown body structure for every node type written by `spec-decomposer` into the Obsidian vault (`vault/tasks/`).

All files MUST use `.md` extension and be stored in `vault/tasks/` (or a dated subdirectory if multiple specs are processed).

---

## YAML Frontmatter Schema (All Node Types)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | Yes | Unique identifier (e.g., `REQ-001-EPIC`, `REQ-001-T01`) |
| `type` | enum | Yes | `epic` \| `story` \| `task` \| `subtask` \| `nfr` \| `security` |
| `parent` | string | Yes | Parent node ID; `root` for epics |
| `title` | string | Yes | Human-readable title (max 80 chars) |
| `status` | enum | Yes | `pending` \| `in-progress` \| `blocked` \| `review` \| `complete` |
| `priority` | enum | Yes | `P0` \| `P1` \| `P2` \| `P3` |
| `source_spec` | string | Yes | Filename of originating spec (e.g., `SPEC.md`) |
| `source_section` | string | Yes | Heading of the spec section containing the requirement |
| `source_requirement` | string | Yes | Requirement ID (e.g., `REQ-001`) |
| `acceptance_criteria` | string[] | Yes | IDs of linked acceptance criteria (min 1 for story/task/subtask) |
| `definition_of_done` | string[] | Yes | Checklist items that define completion (min 1 for story/task/subtask) |
| `dependencies` | string[] | No | Node IDs this node depends on (hard) |
| `blocks` | string[] | No | Node IDs this node blocks |
| `assignee` | string | No | Owner identifier |
| `estimate` | string | Yes | Effort estimate (e.g., `4h`, `2d`) — mandatory for story/task/subtask |
| `gherkin_scenario` | string | No | Full `Given/When/Then` text if applicable |
| `flagged` | boolean | No | `true` if this node was created from an ambiguous requirement or failed validation |
| `adr_references` | string[] | No | IDs of ADRs that constrain or affect this node (e.g., `["ADR-003", "ADR-007"]` ) |
| `inferred_nfrs` | string[] | No | Non-functional requirements inferred by context (e.g., `["Latency < 100ms at p99", "PCI-DSS SAQ-A"]` ) |
| `security_tags` | string[] | No | Security labels auto-applied by inference engine (e.g., `["owasp", "pci-dss", "encryption"]` ) |
| `validation_errors` | string[] | No | Populated when mandatory fields are missing or ADR violations detected |

## Epic Template

Epics represent major feature areas or deliverables. They are the root of a decomposition branch.

```yaml
---
id: REQ-001-EPIC
type: epic
parent: root
title: User Authentication System
status: pending
priority: P0
source_spec: SPEC.md
source_section: "## 3. Authentication"
source_requirement: REQ-001
acceptance_criteria: ["AC-1", "AC-2", "AC-3"]
definition_of_done:
  - All stories under this epic are complete
  - Security audit passed
  - Integration tests green
dependencies: []
blocks: [REQ-001-STORY]
assignee: ""
estimate: "5d"
gherkin_scenario: ""
adr_references: ["ADR-001"]
inferred_nfrs: ["Latency < 100ms for auth endpoints at p99", "OWASP ASVS Level 2"]
security_tags: ["owasp", "auth", "rate-limit"]
---
```

## Flagged / Ambiguous Node Template

When a requirement fails validation, mandatory fields are missing, or an ADR violation is detected, create a blocked node instead of a full decomposition. This preserves traceability without producing low-quality tasks.

```yaml
---
id: REQ-003-EPIC
type: epic
parent: root
title: "[FLAGGED] Implement fast and seamless checkout"
status: blocked
priority: P2
source_spec: SPEC.md
source_section: "## 5. Checkout"
source_requirement: REQ-003
acceptance_criteria: []
definition_of_done: []
dependencies: []
blocks: []
assignee: ""
estimate: ""
gherkin_scenario: ""
flagged: true
adr_references: []
inferred_nfrs: []
security_tags: []
validation_errors:
  - "contains untestable weasel word 'fast'"
  - "contains untestable weasel word 'seamless'"
  - "missing acceptance criteria"
---

# [FLAGGED] Implement fast and seamless checkout

## Traceability

## Source

## Next Steps

2. Add definition of done
```

## NFR / Security / Performance Node Template

Non-functional requirements are first-class nodes. They use the standard templates above but are tagged with `type: nfr`, `type: security`, or `type: performance`.

## Validation Checklist for Manual Review

Before accepting a set of generated task nodes, verify:

- [ ] Every epic has at least one story
- [ ] Every story has at least one task
- [ ] Every task at story level or below has a non-empty `definition_of_done`
- [ ] Every task at story level or below has at least one `acceptance_criteria` entry
- [ ] Every task at story level or below has a non-empty `estimate`
- [ ] No task has an estimate exceeding 8 hours
- [ ] No subtask has an estimate below 15 minutes
- [ ] Every node has bidirectional traceability (`source_spec`, `source_section`, `source_requirement`)
- [ ] `dependencies` and `blocks` are mutually consistent (if A blocks B, B depends on A)
- [ ] Ambiguous requirements are flagged, not silently dropped
- [ ] Inferred NFRs are reviewed for accuracy and adjusted if needed
- [ ] ADR cross-references are populated for tasks that touch architecture boundaries
- [ ] ADR violations are resolved or explicitly accepted with documented exception
- [ ] spec-index.md is present and lists all epics, ambiguities, ADR violations, and inferred NFRs
